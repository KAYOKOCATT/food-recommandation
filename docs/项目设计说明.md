# 项目设计说明

## 1. 项目概述

本项目是一个基于 Django 的餐饮个性化推荐系统，目标不是单独实现某一种推荐算法，而是完成一个从数据、模型、页面到工程边界都可运行、可展示、可解释的毕设系统。

项目最终形成了两条并行但兼容共存的业务线：

- 中文菜品业务线：提供本地用户、菜品、收藏、评论、热门推荐与演示型协同过滤，保证系统具备完整站内业务闭环。
- Yelp 餐厅业务线：基于更标准、更权威的公开数据集承载推荐系统设计与工程落地，重点展示内容推荐、离线构建、近期行为重排和多身份演示链路。

这两条业务线并不是一开始就预设好的“理论最优架构”，而是在项目推进过程中针对数据质量、业务可演示性和推荐效果逐步演化出来的折中结果。

在项目后期，为满足毕设答辩对“大数据组件”的展示要求，系统又补入了一层轻量 Spark 离线批处理能力，但没有推翻原有的 Web 主链路。最终形成的是：

- Django + MySQL 负责在线业务与页面展示
- Python / scikit-learn 负责原有内容推荐构建
- Spark 负责 Yelp 大规模原始数据的离线统计与 ALS 模型实验

## 2. 背景与设计动机

### 2.1 为什么没有只做中文菜品线

中文菜品线最初的优势是：

- 数据获取门槛低，能快速做出页面和业务闭环。
- 适合演示用户注册、登录、收藏、评论、菜品详情等基础 Web 功能。

但它也有明显限制：

- 数据来源更偏爬虫与整理，不够权威，字段质量和覆盖度有限。
- 用户行为数据并不天然存在，需要站内积累或通过脚本补足。
- 更适合作为“业务系统演示”，不适合作为推荐算法主实验载体。

因此，中文菜品线被保留下来，但定位调整为：

- 支撑本地业务展示
- 承担基础推荐能力
- 为毕设提供完整前后端闭环

### 2.2 为什么新增 Yelp 餐厅线

随着项目推进，仅依赖中文菜品数据已经不足以支撑更完整的推荐系统论证，因此额外引入了 Kaggle/Yelp Academic Dataset 这一更标准的数据源。

选择 Yelp 的原因包括：

- 商家、用户、评论、评分结构完整，天然符合推荐系统标准范式。
- 同时具备结构化元数据和文本评论，适合做内容推荐。
- 数据规模和稀疏性更接近真实推荐场景，能暴露更真实的工程问题。

因此，Yelp 业务线承担了项目中“推荐系统设计主线”的角色。

## 3. 核心设计原则

整个项目的设计遵循以下原则：

- 先做可运行成品，再做算法增强。
- 优先保证演示稳定性，不把重计算放进请求链路。
- 不为理论完整性牺牲工程可控性。
- 能统一的数据模型尽量统一，避免维护两套认证和页面逻辑。
- 推荐算法按场景拆分，不追求单一模型覆盖所有问题。
- 大数据组件只进入离线构建层，不让在线请求链路依赖 Spark。

## 4. 业务架构设计

### 4.1 双业务线架构

系统按业务目标拆分为三个 Django app：

- `apps.users`
- `apps.foods`
- `apps.recommendations`

其中：

- `users` 负责统一用户模型、登录认证、会话身份、导航注入、演示账号等横切能力。
- `foods` 负责中文菜品业务。
- `recommendations` 负责 Yelp 数据模型、推荐服务、图表和离线构建命令。

这意味着项目不是“两个独立网站拼在一起”，而是“一个统一认证与页面体系下的双业务域系统”。

### 4.2 为什么坚持统一入口而不是拆两个系统

如果把中文菜品与 Yelp 分成两个完全独立系统，会带来额外复杂度：

- 两套用户与登录体系
- 两套导航和权限逻辑
- 两套页面骨架
- 两套部署与演示入口

因此项目最终采用了统一站点、统一用户模型、统一会话体系的方式，让两条业务线共享基础设施，但在业务数据和推荐链路上保持相对独立。

## 5. 数据模型设计

## 5.1 统一用户模型

项目没有为 Yelp 单独再建一张用户主表，而是扩展 `apps.users.models.User`：

- `source`：标记用户来源，区分 `local` 与 `yelp`
- `external_user_id`：记录外部数据源中的用户 ID

这样做的意义在于：

- 本地注册用户与导入用户可以共用同一套会话与权限逻辑
- 页面模板和导航不需要分叉两套“用户身份树”
- Yelp 评论和本地行为可以统一关联到 `User`

这是双业务线兼容共存中最关键的一步，也是实现成本最高的一步之一。

## 5.2 中文菜品模型

中文菜品业务围绕以下表展开：

- `Foods`
- `Collect`
- `Comment`

其中：

- `Foods.collect_count`、`Foods.comment_count` 作为展示与统计排序字段。
- `Collect` 作为协同过滤的隐式反馈源。
- `Comment` 目前仍保留较轻量的历史结构，避免在毕业设计阶段引入大规模数据迁移。

## 5.3 Yelp 模型

Yelp 业务线围绕以下表展开：

- `YelpBusiness`
- `YelpReview`

关键设计点：

- `YelpBusiness.review_count` 保留原始 Yelp 元数据中的评论数。
- `YelpBusiness.aggregated_review_count` 表示当前系统内实际存储的评论总数。
- `YelpReview.source` 区分导入 Yelp 评论与站内新增评论。

这样做的原因是：

- 原始数据热度和系统内可见评论量不是同一个概念。
- 在线展示、推荐重排和离线重建都需要知道“系统内真实可用评论量”。
- 站内本地评论不能覆盖掉导入评论，也不能与原始元数据混淆。

## 6. 前端与交互架构

项目前端采用双轨策略：

- 传统 Django 模板为主
- Alpine.js + HTMX 作为局部增强

模板基座分为：

- `layout.html`：传统后台布局
- `base_auth.html`：登录/注册等无侧边栏页面
- `base_modern.html`：内部现代页面
- `base_chart.html`：图表页面

设计原因：

- 现有模板资源和后台 UI 可以快速支撑页面成型。
- 登录页、局部交互和图表页适合用轻量增强，而不是重写为前后端分离。
- 毕设目标是“完整系统落地”，不是证明某个前端框架本身。

## 7. 登录与身份设计

系统支持三种登录入口：

- 本地账号登录
- Yelp 演示登录
- 管理员演示登录

会话态统一由 `apps.users.session_auth` 管理，核心字段包括：

- `auth_role`
- `login_source`
- `is_demo_login`

这套设计解决了两个问题：

### 7.1 演示身份与真实本地身份共存

Yelp 导入用户并不适合作为常规注册用户去操作完整站内业务，但它们又是推荐演示的重要载体。因此系统没有直接把“Yelp 用户 = 普通用户”，而是通过 `login_source=yelp_demo` 显式标记其演示身份。

### 7.2 页面和权限按身份动态收敛

不同身份会影响：

- 默认跳转页面
- 左侧菜单注入
- 是否可见 Yelp 个性化推荐入口
- 是否可进入管理员后台

这让系统在一个站点内同时容纳“本地业务用户”“推荐演示用户”“后台管理员”成为可能。

## 8. 推荐系统设计

## 8.1 中文菜品推荐链路

中文菜品线的推荐设计偏务实，目标是支撑页面和演示，而不是追求严格学术评测。

当前主要包括：

- 热门统计推荐
- 基于收藏的 ItemCF / UserCF

### 设计考虑

- 中文菜品本身没有天然的大规模用户交互数据。
- 若完全等待站内真实行为累积，毕设周期内难以形成可展示的协同过滤效果。
- 因此引入 `generate_demo_collects` 命令生成明确标注的演示收藏数据。

这一点必须明确说明：

- 这些收藏是 synthetic implicit feedback
- 它们用于演示协同过滤链路是否可跑通
- 它们不作为真实效果评估依据

也就是说，中文菜品线的 CF 更偏“工程演示能力”，而不是“真实线上效果论证”。

## 8.2 Yelp 内容推荐链路

Yelp 业务线优先落地的是内容推荐。

原因包括：

- 商家元数据完整，适合做 item profile
- 评论文本丰富，适合做 TF-IDF
- 内容相似度可以离线预计算
- 对新餐厅和弱行为用户更稳定

因此，项目将 `build_yelp_content_recs` 作为 Yelp 主推荐基础设施之一，生成：

- `yelp_content_itemcf.json`
- `yelp_business_profiles.json`

这里的命名保留了“itemcf”字样，但实际推荐逻辑更接近“内容相似 item-to-item 候选”。

## 8.3 为什么 Yelp UserCF 没有成为主链路

项目中确实实现了评分版 Yelp UserCF，并保留了 `build_yelp_review_usercf` 命令，但它最终没有进入主页面核心链路。

原因有三类：

### 1. 数据稀疏性问题

Yelp 数据在推荐任务上表现出明显稀疏特征。探索阶段观察到：

- 单用户可用于建模的餐厅覆盖度偏低
- 用户间共同评分的交集较小
- 在邻域法中，很容易出现“找不到足够稳定邻居”的问题

在项目讨论中，这一问题甚至具体表现为“平均单用户评价餐厅数偏低，约在 1.x 的量级口径下波动”，这直接削弱了 UserCF 的有效性。这里的数值会随筛选口径、样本集规模和展示集/构建集边界而变化，但结论是一致的：用户维度比餐厅维度更稀疏、更不稳定。

### 2. 工程代价与收益不匹配

UserCF 的问题不仅是效果，还包括：

- 构建规模随用户数增长迅速膨胀
- 共同评分阈值、活跃度边界、构建集规模都需要额外控制
- 运行时若依赖 UserCF，页面稳定性和可解释性都更差

因此项目只能把它收敛为：

- 离线实验能力
- 可供对比或扩展的备用链路
- 不进入主页面强依赖

### 3. 业务场景更适合“最近行为 + 物品相似”

对餐厅推荐页面而言，用户最近评分过什么、这些餐厅附近有哪些内容相似候选，往往比“相似用户喜欢了什么”更直接、更稳定、更容易解释。

这也是后续提交中从 “Yelp UserCF 主导” 迁移到 “近期评论 + 内容邻域 + 热门兜底” 的根本原因。

## 8.4 为什么新增 Spark ALS 但不替换主链路

项目后期补入 Spark ALS，并不是因为现有默认推荐链路不可用，而是出于两个现实原因：

- Yelp 原始数据在量级上已经具备“大规模离线批处理”的论证空间
- 毕设答辩需要更明确的大数据组件落地痕迹

之所以没有直接让 ALS 取代默认推荐页，是因为：

- ALS 更偏“模型实验能力”，不一定比当前内容链路更稳定
- 当前内容推荐 + 最近行为重排的解释性更强，更适合演示
- 若把默认页面完全切到 ALS，会显著提高线上依赖和排障成本

因此，系统对 Spark ALS 的定位被收敛为：

- Spark 离线训练产物
- 独立的 Yelp ALS 实验推荐页
- 用于答辩展示“大数据推荐建模能力”

而默认 Yelp 推荐页仍保持：

- 最近评分行为提取
- 离线内容相似候选重排
- 热门兜底

## 8.5 Yelp 主推荐链路的最终形态

Yelp 主链路最终采用三段式：

- 第一层：最近评分行为提取
- 第二层：读取离线内容相似候选并做轻量重排
- 第三层：若近期行为不足，则回退到热门推荐

对应实现是 `YelpService.get_recent_recommendations`：

- 先读取用户最近评分过的餐厅
- 再调用 `rerank_from_recent_items`
- 最后用热度做轻量 blended score

这种设计的优势是：

- 运行时简单
- 结果可解释
- 可以充分复用离线内容相似度文件
- 对演示账号更加稳定

## 9. 离线构建与在线服务边界

这是本项目最重要的工程取舍之一。

### 9.1 为什么要把重计算移出请求链路

如果在线页面直接承担以下计算：

- 相似度构建
- 全量用户邻域搜索
- 评论文本聚合
- 演示账号活跃度统计

会导致：

- 页面响应慢
- 数据规模上升后体验不稳定
- 演示过程中结果不可控

因此，项目通过管理命令把重计算前移到离线阶段。

### 9.2 关键离线产物

项目当前主要依赖以下离线产物：

- `food_itemcf.json`
- `food_usercf.json`
- `yelp_content_itemcf.json`
- `yelp_usercf.json`
- `yelp_als_userrec.json`
- `yelp_spark_hot.json`
- `yelp_spark_city_top.json`
- `yelp_spark_monthly_stats.json`
- `yelp_demo_users.json`
- 首页词云图片

### 9.3 离线演示账号候选的必要性

登录页一开始如果实时聚合全量 Yelp 用户和评论，会让首页承担大量无意义查询。为此项目专门增加了：

- `refresh_yelp_demo_users`
- `apps.users.demo_candidates`

登录页改为直接读取离线候选 JSON，而不是在请求链路里统计所有 Yelp 用户的活跃度。

这属于典型的“为演示稳定性做架构收敛”。

## 10. 从 notebooks 到离线产物再到 Web 的落地流程

项目中的 notebook 并不是直接被 Web 调用的“生产代码”，它们的角色更接近：

- 进行数据探索
- 验证数据集是否适合推荐任务
- 确定特征选择和算法方向
- 为后续 Django 命令和服务层提供方法论依据

也就是说，notebook 负责“找方向”，真正上线的离线产物则由管理命令和服务层代码生成。

### 10.1 总体落地链路

核心链路可以概括为：

1. `notebooks/` 中完成数据探索与方案验证
2. 将验证过的方法固化到 Django management command
3. 命令输出 JSON / PNG 等离线产物，或导入 MySQL
4. Web 层通过 service 读取这些产物
5. view 将结果交给模板或 API 输出

这条链路的意义是把“实验结论”转换成“可稳定调用的工程资产”。

### 10.2 Yelp 内容推荐产物的落地

这是最典型的一条从 notebook 走向 Web 的路径。

#### notebook 阶段

`notebooks/README.md` 与相关探索 notebook 的主要结论是：

- `archive_4` 的 Yelp 数据集最适合作为推荐主线
- restaurant business + review text 的组合适合做内容推荐
- TF-IDF + 物品相似度比一开始就上复杂模型更适合当前工程阶段

#### 工程固化阶段

上述结论最终被固化到：

- `apps/recommendations/yelp_content.py`
- `apps/recommendations/management/commands/build_yelp_content_recs.py`

具体代码流程如下：

1. `build_yelp_content_recs` 读取 `data/archive_4/` 下的 `business` 和 `review` JSONL 文件。
2. 命令调用 `iter_json_lines` 流式读取原始数据，避免一次性加载大文件。
3. `build_business_profiles` 先用 `is_restaurant_business` 过滤出适合当前建模的餐厅：
   - 必须是餐厅类业务
   - 过滤明显不适合的类别
   - 默认过滤关闭商家
   - 要求有经纬度
   - 要求达到最小评论数阈值
4. 对每个保留下来的餐厅，聚合有限数量的评论文本，形成 `YelpBusinessProfile`。
5. `YelpBusinessProfile.combined_text()` 将名称、类别、评论文本合并为一个文档。
6. `build_yelp_content_recommendations()` 用 `TfidfVectorizer` 将每家餐厅转成 TF-IDF 向量。
7. 再用 `NearestNeighbors(metric="cosine")` 为每家餐厅找 top-k 相似邻居。
8. 结果通过 `serialize_similarity()` 写入 `data/recommendations/yelp_content_itemcf.json`。
9. 同时把可检查的餐厅元数据写入 `data/recommendations/yelp_business_profiles.json`。

#### Web 消费阶段

这个离线 JSON 最终进入 Web 的路径有三条：

##### 1. 餐厅详情页相似推荐

- `apps.recommendations.views.yelp.yelp_business_detail`
- `YelpService.get_similar_businesses`
- `similarity_cache.get(...)`

流程是：

1. 详情页请求进入 `yelp_business_detail`
2. view 调用 `YelpService.get_similar_businesses`
3. service 从 `yelp_content_itemcf.json` 读取当前商家的候选邻居
4. 再回查 ORM，把 JSON 中的 `business_id` hydrate 成 `YelpBusiness`
5. 模板 `templates/recommendations/yelp_business_detail.html` 展示相似餐厅

##### 2. Yelp 为你推荐页

- `apps.recommendations.views.yelp.yelp_recommendations`
- `YelpService.get_recent_recommendations`
- `rerank_from_recent_items`

流程是：

1. 先从 `YelpReview` 找出当前用户最近评分过的餐厅
2. 再把这些餐厅 ID 当作种子，读取 `yelp_content_itemcf.json`
3. `rerank_from_recent_items` 按“越近的行为权重越高”合并候选
4. `YelpService` 再叠加一个轻量的 popularity score
5. 最终输出给 `templates/recommendations/yelp_recommendations.html`

也就是说，Yelp 个性化推荐页虽然看起来像“在线推荐”，本质上仍然是在离线内容相似候选上做在线轻量重排。

##### 3. 相似度网络图

- `apps.recommendations.services.chart_service.ChartService.get_similarity_network`
- `apps.recommendations.views.charts.ChartView.similarity_network`

流程是：

1. 图表 API 从 `yelp_content_itemcf.json` 读取相似度映射
2. 按阈值筛选边
3. 再通过 ORM 补齐节点名称、分类和评论数
4. 组装为 ECharts 网络图所需的 `nodes` 和 `links`

因此，同一份离线相似度文件同时服务于：

- 详情页相似餐厅
- 推荐页候选召回
- 可视化网络图

这就是离线产物复用的核心价值。

### 10.3 Spark 统计与 ALS 产物的落地

为尽量减少对现有系统的侵入，Spark 能力没有直接进入请求链路，而是固化为两类离线作业：

- `apps/recommendations/spark_jobs/build_stats.py`
- `apps/recommendations/spark_jobs/build_als.py`

并通过 Django 管理命令封装为：

- `build_yelp_spark_stats`
- `build_yelp_spark_als`

其中，`build_yelp_spark_stats` 的职责是：

1. 直接读取 Yelp 原始 JSONL
2. 用 Spark SQL 聚合热门餐厅、城市榜单、月度评论趋势
3. 写入 `yelp_spark_hot.json`
4. 写入 `yelp_spark_city_top.json`
5. 写入 `yelp_spark_monthly_stats.json`

ALS 链路则是：

1. 从 `YelpReview` 导出用户-餐厅-评分交互
2. 调用 Spark ALS 训练矩阵分解模型
3. 生成每个用户的 top-k 餐厅推荐
4. 写入 `yelp_als_userrec.json`
5. 在线由 `YelpService.get_als_recommendations` 读取
6. 最终展示到独立的 Yelp ALS 实验推荐页

这条链路的定位不是“替代默认推荐页”，而是：

- 补齐答辩中的大数据组件落地
- 补齐模型型协同过滤能力
- 保持线上默认推荐页的稳定性

### 10.4 中文菜品协同过滤产物的落地

中文菜品线没有原生的大规模交互数据，因此这条链路和 Yelp 不同。

#### notebook / 思路阶段

中文菜品部分没有像 Yelp 那样依赖大型公开评分数据集，更多是基于系统现有业务模型去思考：

- 用户收藏可以视作隐式反馈
- 只要有足够多的 `Collect(user_id, food_id)`，就可以构造 ItemCF / UserCF

#### 工程固化阶段

对应命令有两个：

- `apps/foods/management/commands/generate_demo_collects.py`
- `apps/foods/management/commands/build_food_collect_cf.py`

代码流程如下：

1. `generate_demo_collects` 先按用户偏好菜系和一定探索率生成演示收藏。
2. 这些收藏写入 `Collect` 表，并同步增加 `Foods.collect_count`。
3. 然后 `build_food_collect_cf` 从数据库读取 `(user_id, food_id)` 交互对。
4. 命令调用 `apps.recommendations.collect_cf` 中的：
   - `item_cf_similarities`
   - `user_cf_recommendations`
5. 最终输出：
   - `food_itemcf.json`
   - `food_usercf.json`

#### Web 消费阶段

这些文件最终由 `apps.foods.services` 读取：

- `similar_foods_for_detail` 读取 `food_itemcf.json`
- `recommend_foods_by_usercf` 读取 `food_usercf.json`
- `recommend_foods_by_itemcf` 也复用了 `rerank_from_recent_items` 的“最近行为重排”思路

这条链路说明：

- 中文菜品 CF 的核心不是 notebook 推导出复杂模型
- 而是把现有业务表转成隐式反馈矩阵，再生成足够稳定的演示型离线产物

### 10.5 首页词云产物的落地

词云虽然不是推荐结果本身，但它非常能代表本项目的“离线特征加工 -> 在线展示”思路。

#### 思路阶段

探索阶段已经验证：

- 中文菜品的 `Foods.recommend` 可以作为轻量中文文本源
- Yelp 评论文本可以做 TF-IDF 关键词提取

#### 工程固化阶段

对应模块是：

- `apps.recommendations.management.commands.build_home_wordclouds`
- `apps.recommendations.services.home_wordcloud_service.HomeWordCloudService`

代码流程如下：

##### 中文词云

1. 从 `Foods.recommend` 读取非空推荐语
2. 用 `_tokenize_food_text` 做中文/英文 token 提取
3. 统计词频
4. 调用 `WordCloud.generate_from_frequencies`
5. 输出 `home_food_recommend_wordcloud.png`

##### Yelp 评论词云

1. 从 `YelpReview.text` 读取文本
2. 用 `TfidfVectorizer` 提取英文关键词
3. 取 top words 及其权重
4. 调用 `WordCloud.generate_from_frequencies`
5. 输出 `home_yelp_review_wordcloud.png`

#### Web 消费阶段

首页相关流程如下：

1. `apps.users.views.user_index` 用 `HomeWordCloudService.image_exists()` 判断词云是否存在
2. 模板根据可用性决定是否展示入口
3. `home_wordcloud_image` 通过 `HomeWordCloudService.get_image_path()` 返回 PNG 文件

也就是说，首页本身不做任何词云计算，只负责读取离线图片。

### 10.6 Yelp 演示账号产物的落地

这也是一个典型“从数据加工到页面可用”的离线产物。

#### 设计动机

如果登录页实时扫描全量 Yelp 用户与评论，只是为了给一个下拉框列出演示账号，会让首页承担不必要的大查询。

#### 工程固化阶段

对应模块是：

- `apps.users.management.commands.refresh_yelp_demo_users`
- `apps.users.demo_candidates`

代码流程如下：

1. 从 `User(source="yelp")` 出发
2. 聚合每个用户的 `review_count` 与 `last_review_at`
3. 过滤掉没有评论的导入用户
4. 按活跃度排序后截断
5. 输出 `yelp_demo_users.json`

#### Web 消费阶段

1. 登录页 GET 请求时调用 `_get_yelp_demo_users()`
2. 实际底层通过 `load_yelp_demo_candidates()` 读取 JSON
3. 模板 `auth/login-refactored.html` 渲染 Yelp 演示账号候选
4. `login_yelp_demo` 只允许候选列表中的账号走演示登录链路

这条链路体现的不是推荐算法，而是“离线筛选 -> 在线轻量消费”的同一设计哲学。

### 10.7 关键结论

从 notebook 到离线产物再到 Web，本项目遵循的是统一模式：

- notebook 负责证明“值得做什么”
- management command 负责把方法变成稳定产物
- JSON / PNG / DB 负责承载可复用结果
- service 负责读取产物并与 ORM 拼接
- view / template 负责最终展示

这比“直接把 notebook 逻辑复制到 view 里”多了一层工程化收敛，也正是本项目从实验走向成品的关键一步。

## 11. Yelp 导入策略设计

Yelp 数据导入并不是简单“读取前 N 行”。

后期项目已从粗暴采样调整为“展示集 + 构建集”语义：

- 先筛出目标餐厅集合
- 再从目标餐厅评论中反推用户集合
- 最后导入这些目标餐厅、目标用户及关联评论

这样做的目的有三点：

- 保证导入数据在业务上可用，不是随机碎片
- 控制数据库规模和构建时间
- 为后续内容推荐与演示账号筛选保留有效信号

此外，导入过程还做了以下兼容工作：

- 通过 `external_user_id` 和稳定生成的 `username` 保证导入幂等
- 评论导入后仅增量刷新受影响餐厅的 `aggregated_review_count`
- 导入完成后顺带刷新 Yelp 演示账号候选

## 12. 图表与可视化设计

图表能力由 `ChartService` 支撑，主要作用不是“炫技”，而是为项目补足两个能力：

- 让推荐系统之外的统计特征有可观察出口
- 证明系统不仅有推荐页面，也有分析能力和数据支撑层

首页词云同样体现了这一设计思想：

- 词云不是在线临时生成
- 而是离线计算后在线读取图片

这延续了项目一贯的离线优先策略。

## 13. 测试与鲁棒性设计

项目虽然不是工业系统，但在关键边界上做了针对性测试：

- Yelp 导入幂等性
- 非餐厅数据过滤
- 评论导入后的聚合评论数刷新
- 相似度 JSON 缺失、损坏、候选缺失时的降级
- 推荐接口在近期行为缺失时的热门兜底
- 多种身份会话下的页面访问逻辑

这意味着项目不是只完成“主流程 happy path”，而是对关键降级场景有明确处理。

## 15. 设计亮点

本项目的亮点主要不在“模型有多先进”，而在“工程取舍清晰”：

- 双业务线不是简单拼接，而是统一用户模型和页面体系下的兼容共存。
- 中文菜品线与 Yelp 线分工明确，各自承担不同毕设目标。
- 能离线的计算尽量离线，在线只保留查询、过滤和轻量重排。
- 明确区分真实业务数据、导入数据和演示数据。
- 对推荐主链路做了效果与工程成本之间的折中，而不是盲目坚持 UserCF。

## 16. 局限与后续优化方向

当前设计仍有局限：

- 中文菜品协同过滤依赖演示收藏数据，缺少真实长期行为支撑。
- Yelp UserCF 虽保留，但没有形成足够强的主链路价值。
- 离线评测体系仍可继续补充，例如 Precision@K、Recall@K。
- 当前推荐解释更多依赖工程描述，而非显式解释模块。

后续可以考虑：

- 增强离线评测与实验记录
- 引入更细粒度的 rerank 特征
- 进一步统一本地评论与 Yelp 评论在推荐中的权重策略
- 将设计说明、答辩稿、实验结论形成更完整的 docs 体系

## 17. 总结

本项目的最终价值不在于“把所有推荐算法都实现一遍”，而在于完成了一次现实条件下的推荐系统工程落地：

- 数据不完美
- 业务目标不单一
- 页面演示需要稳定
- 模型效果与工程复杂度必须权衡

在这样的约束下，项目最终形成了“双业务线 + 统一用户体系 + 离线优先 + 轻量在线重排”的架构方案。这一方案既保住了中文菜品业务闭环，也让 Yelp 数据承担了更完整的推荐系统设计主线，是目前这个毕设条件下较为合理的落地结果。
