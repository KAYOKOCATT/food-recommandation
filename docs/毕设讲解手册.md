# 餐饮个性化推荐系统讲解手册

## 1. 讲解目标

这份手册用于向同学或答辩老师完整讲解本项目的设计核心。讲解重点不是逐个文件念代码，而是讲清楚四件事：

- 这个毕设想解决什么问题
- 为什么系统会设计成现在这样
- 推荐算法是怎样落地到 Web 系统里的
- 项目是如何从实验想法逐步迭代成可演示成品的

## 2. 项目一句话概括

这是一个基于 Django 的餐饮个性化推荐系统。它同时包含：

- 中文菜品业务线：负责站内用户、菜品、收藏、评论和基础推荐闭环
- Yelp 餐厅业务线：负责展示更完整的推荐系统建模、离线构建和个性化推荐流程

## 3. 选题背景与问题定义

可以先从业务问题切入：

- 用户面对大量菜品或餐厅时，选择成本高
- 单纯展示热门内容不够个性化
- 推荐系统既要能演示算法，也要能真正接到页面上

因此本项目的目标不是只做一个算法 notebook，而是做一个“能跑、能看、能解释”的推荐系统成品。

## 4. 为什么拆成两条业务线

### 中文菜品线

作用是先把站内业务闭环做完整：

- 用户注册、登录
- 菜品列表与详情
- 收藏与评论
- 热门推荐与协同过滤演示

这一条线更像“系统基础盘”，保证毕设不是只有离线实验。

### Yelp 餐厅线

作用是承载更标准的推荐系统问题：

- 有商家元数据
- 有用户评论和评分
- 有更适合做内容推荐与协同过滤的数据结构

这一条线更像“推荐能力展示盘”，体现算法和工程结合。

## 5. 系统总体架构

讲解时建议按这一条主线说：

1. 前端页面用 Django 模板渲染
2. 视图层接收请求并调用服务层
3. 服务层从 MySQL 和离线 JSON 中读取数据
4. 推荐结果返回页面展示

系统核心模块：

- `apps.users`：认证、会话身份、个人中心、演示账号
- `apps.foods`：中文菜品、收藏、评论、统计推荐、收藏协同过滤
- `apps.recommendations`：Yelp 数据模型、推荐服务、图表、离线构建命令

## 6. 数据模型设计核心

### 用户模型统一

本地用户和 Yelp 导入用户统一使用 `users.User`，而不是维护两套用户表。

这样做的原因：

- 登录逻辑统一
- 页面访问控制统一
- 推荐入口统一
- 减少后期联表和权限判断复杂度

### 中文菜品表

- `Foods`
- `Collect`
- `Comment`

这三张表支撑中文菜品的站内行为闭环。

### Yelp 表

- `YelpBusiness`
- `YelpReview`

`YelpReview` 同时承载导入的 Yelp 评论和站内新增评论，通过 `source` 区分来源。

## 7. 推荐系统设计分层

项目不是只用一种算法，而是按场景拆分。

### 中文菜品推荐

- 热门统计推荐：按收藏数、评论数等排序
- 收藏协同过滤：基于 `Collect` 生成 ItemCF/UserCF 离线 JSON

### Yelp 推荐

- 内容推荐：基于 business 信息和 review 文本做 TF-IDF 相似度
- UserCF 实验：离线生成用户到餐厅的候选结果
- 近期行为重排：读取用户最近评分过的餐厅，再对离线相似候选做重排
- 热门兜底：没有近期行为时直接回退到热门商家

## 8. 为什么采用“离线构建 + 在线读取”

这是本项目最值得重点讲的工程取舍之一。

如果把相似度计算、用户邻域计算、词云统计都放在请求链路里，会有三个问题：

- 查询慢
- 页面不稳定
- 演示时不可控

所以项目采用：

- 离线命令负责重计算
- 在线页面只读取数据库和 JSON 产物

主要离线产物包括：

- `food_itemcf.json`
- `food_usercf.json`
- `yelp_content_itemcf.json`
- `yelp_usercf.json`
- `yelp_demo_users.json`
- 首页词云图片

## 9. 登录与权限设计

系统支持三类入口：

- 本地账号登录
- Yelp 演示登录
- 管理员演示登录

会话中通过以下字段统一判断身份：

- `auth_role`
- `login_source`
- `is_demo_login`

这一设计的价值在于：

- 同一套用户系统可以承载不同来源身份
- 不同身份可以看到不同菜单与入口
- 推荐页面和后台页面的访问控制更清晰

## 10. 前端架构怎么讲

前端不是完全重写，而是保留原模板体系并做渐进增强。

- `layout.html`：传统后台布局
- `base_auth.html`：登录、注册等页面
- `base_modern.html`：内部现代页面
- `base_chart.html`：图表页面

增强技术：

- Alpine.js：轻量交互
- HTMX：局部请求增强
- ECharts：图表展示

这部分要强调：本项目是“传统模板为主，现代增强为辅”，不是前后端分离项目。

## 11. 页面演示顺序建议

讲解时推荐按下面顺序：

1. 登录页
2. 普通用户首页
3. 中文菜品列表与详情
4. 中文菜品推荐页
5. Yelp 餐厅发现页
6. Yelp 餐厅详情页
7. Yelp 为你推荐页
8. 数据可视化页
9. 管理员后台页

这样讲的好处是从“基础业务”逐步过渡到“推荐能力”和“工程化能力”。

## 12. 项目演进过程

可以结合 git 提交记录讲成五个阶段：

### 第一阶段：站内业务打底

- 完成用户、菜品、评论、收藏、基础页面

### 第二阶段：引入 Yelp 数据

- 做 Yelp 数据探索
- 完成商家与评论落库
- 打通餐厅列表和详情页

### 第三阶段：补充推荐能力

- 加入内容推荐
- 加入 UserCF 等离线实验能力

### 第四阶段：做工程化收敛

- 把运行时重查询迁出请求链路
- 改成离线 JSON 驱动
- 加入词云等离线展示产物

### 第五阶段：统一身份与演示入口

- 三类登录入口收敛
- 菜单和权限按身份控制
- 登录页不再实时聚合全量 Yelp 用户

## 13. 项目亮点

- 同时覆盖 Web 系统开发和推荐系统工程落地
- 中文菜品线和 Yelp 线分工明确
- 离线计算和在线服务边界清晰
- 页面、推荐、图表、数据导入形成完整闭环
- 推荐逻辑具备一定可解释性，适合答辩展示

## 14. 项目局限

这部分建议主动讲，反而更显得设计成熟：

- 当前主页面没有采用更复杂的矩阵分解或深度学习模型
- Yelp UserCF 主要保留为实验能力，不是主链路
- 离线评测体系可以继续补强
- 推荐结果的线上用户反馈闭环还不够完整

## 15. 后续可扩展方向

- 增加离线评测指标，如 Precision@K、Recall@K
- 引入更细粒度的 rerank 特征
- 为用户建立更稳定的兴趣画像
- 把更多推荐实验结果沉淀成可复用服务模块
- 补充更系统的文档与答辩材料

## 16. 结尾总结模板

最后可以用一句话收尾：

“这个项目的核心不在于单独实现某一种推荐算法，而在于把推荐系统从数据、模型、页面到工程边界完整落成一个可运行、可展示、可解释的毕设系统。”
