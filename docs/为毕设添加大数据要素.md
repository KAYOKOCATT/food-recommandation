## 一、大数据推荐系统整体架构

### 1.1 传统系统 vs 大数据系统

| 维度 | 传统系统 | 大数据系统（目标） |
|------|---------------------|-------------------|
| **数据存储** | MySQL/PostgreSQL | HDFS + HBase/MongoDB |
| **计算引擎** | Python单进程/Pandas | Spark分布式计算 |
| **实时处理** | 难以实现 | Kafka + Spark Streaming/Flink |
| **数据量级** | 百万级 | 亿级甚至更高 |
| **算法训练** | scikit-learn本地训练 | Spark MLlib分布式训练 |

### 1.2 大数据处理流程（数据生命周期）

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   数据源     │ → │   数据采集   │ → │   数据存储   │ → │   数据计算   │ → │   数据应用   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
      │                  │                  │                  │                  │
      ▼                  ▼                  ▼                  ▼                  ▼
  业务数据库         ETL工具          HDFS(文件)        MapReduce        实时推荐
  日志文件           Flume            HBase(列式)       Spark            离线推荐
  用户行为           Kafka            Cassandra         Flink            可视化
  外部数据           Sqoop            MongoDB           Storm            BI分析
```

**核心组件速览**：

| 组件 | 作用 | 类比理解 |
|------|------|----------|
| **HDFS** | 分布式文件存储 | 类似云盘，数据分块存储在多台机器 |
| **HBase** | 列式数据库 | 适合存储海量稀疏数据（如用户评分矩阵）|
| **Spark** | 分布式计算引擎 | 把Python代码分布到集群并行执行 |
| **Kafka** | 消息队列 | 数据的"高速公路"，连接各系统 |
| **Flume** | 日志采集 | 自动收集分散的日志文件 |

SELECT rid, COUNT(rid) as count 
FROM ratings 
GROUP BY rid 
ORDER BY count DESC

#### 2）近期热门餐厅统计（按月份）

```sql
-- 先转换时间戳为年月格式
WITH rating_of_month AS (
    SELECT rid, score, 
           DATE_FORMAT(FROM_UNIXTIME(timestamp), 'yyyy-MM') as yearmonth 
    FROM ratings
)
SELECT rid, COUNT(rid) as count, yearmonth 
FROM rating_of_month 
GROUP BY yearmonth, rid 
ORDER BY yearmonth DESC, count DESC
```

## 五、离线推荐模块（ALS算法）

### 5.1 ALS算法原理

ALS（交替最小二乘法）是协同过滤的经典算法：

```
评分矩阵 R (N用户 × M餐厅) ≈ 用户矩阵 U (N × K) × 餐厅矩阵 P (K × M)

其中 K 是隐因子维度（通常50-200）

目标：最小化 ||R - U×P||² + λ(||U||² + ||P||²)
```

**Spark MLlib实现**：

```python
from pyspark.mllib.recommendation import ALS, Rating

# 准备训练数据
ratings_rdd = ratings_df.rdd.map(lambda row: Rating(row.uid, row.rid, row.score))

# 训练ALS模型
model = ALS.train(
    ratings=ratings_rdd,
    rank=50,          # 隐因子维度
    iterations=10,    # 迭代次数
    lambda_=0.01      # 正则化参数
)

# 保存模型
model.save(sc, "hdfs://localhost:9000/models/als_model")
```

### 6.2 推荐优先级计算公式

当用户评分了餐厅A、B、C后，如何计算候选餐厅X的推荐分数？

```
Score(X) = [Σ(sim(i,X) × rating_i)] / K + log(max(1, count)) - log(max(1, count - K))

其中：
- sim(i,X)：餐厅i与X的相似度（从离线计算的相似度矩阵获取）
- rating_i：用户对餐厅i的评分
- K：用户最近评分次数（通常取10-20）
- count：用户总评分次数
- log项：偏移项，鼓励用户探索新餐厅
```

## 九、关键问题Q&A

### Q1：ALS模型多久更新一次？

**A**：取决于数据量和业务需求：
- 小数据量（< 100万评分）：每小时更新
- 中数据量（100万-1亿）：每天更新
- 大数据量（> 1亿）：每周更新

实时推荐弥补离线模型的延迟问题。

### Q2：冷启动问题如何解决？

**A**：
1. **新用户**：展示热门榜单 + 基于注册信息的类别推荐
2. **新餐厅**：内容推荐（基于类别、标签、描述）+ 给予一定曝光机会
3. **混合策略**：协同过滤 + 内容推荐 + 热门推荐的加权组合

### Q3：Spark和Flink如何选择？

**A**：
- **Spark**：批处理为主，流处理为微批（延迟秒级），生态丰富
- **Flink**：纯流处理（延迟毫秒级），实时性要求极高时选用
- **建议**：推荐系统用Spark即可，实时性要求高的场景（如实时风控）用Flink

### Q4：数据存储如何选型？

**A**：

| 数据类型 | 推荐存储 | 原因 |
|---------|---------|------|
| 原始日志 | HDFS | 大文件存储，成本低 |
| 用户评分 | HBase/MongoDB | 随机读写，稀疏矩阵友好 |
| 推荐结果 | MongoDB/Redis | 快速查询，JSON格式灵活 |
| 相似度矩阵 | HBase | 大规模键值对存储 |
| 实时缓存 | Redis | 亚毫秒级响应 |
